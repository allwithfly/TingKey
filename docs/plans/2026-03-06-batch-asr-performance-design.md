# Batch ASR Performance Design

**Date:** 2026-03-06

**Goal**

优先优化批量转写场景的总体耗时，减少 CLI 与常驻 Shell 在多文件输入时的重复串行开销，同时保持现有时间戳回退行为和 GUI / API 的稳定性。

## Current Pain Points

- `asr_cli.py` 对多文件输入逐个执行 `engine.transcribe(...)`，没有利用 `speech_output.py` 已支持的批量音频输入能力。
- `asr_service_shell.py` 也采用逐文件串行转写，虽然复用了已加载模型，但未利用批处理能力。
- `asr_media_utils.py` 中的视频抽音频逻辑不会复用已生成的 WAV，重复转写同一视频时会重复调用 FFmpeg。
- 时间戳请求依赖 forced aligner，当前通过逐次调用和回退保持兼容；这段逻辑不能为了性能优化而被破坏。

## Recommended Approach

采用低风险、可渐进落地的方案：

1. 在共享层补充“批量准备 + 分批推理”辅助能力。
2. 仅在**不需要时间戳**时启用真正的批量推理。
3. 对**需要时间戳 / 字幕**的链路保留现有串行逻辑，以避免破坏 forced aligner 缺失时的兼容回退。
4. 为视频抽音频增加基于输出文件存在性和修改时间的轻量缓存，避免同一会话或重复命令中的重复提取。

## Scope

**本轮纳入：**

- `asr_media_utils.py`
- `asr_cli.py`
- `asr_service_shell.py`
- 新增轻量测试文件，覆盖新增共享逻辑

**本轮暂不纳入：**

- `desktop_gui_app.py` 的大范围重构
- `desktop_service/` 的执行管线统一
- 更激进的并发流水线（媒体准备 / 探测 / 推理重叠执行）

## Design Details

### 1. 批量转写入口

在共享工具层新增批处理辅助函数，负责：

- 接收已准备好的音频条目列表
- 按 `max_inference_batch_size` 切片
- 调用一次 `engine.transcribe(audio=[...])`
- 将返回结果按顺序映射回输入条目

该能力只用于 `return_time_stamps=False` 的路径。

### 2. 媒体准备结果结构化

为 CLI / Shell 共享一组结构化的“待转写条目”信息，至少包含：

- 原始媒体路径
- 供模型使用的音频路径
- 是否由视频抽取得到
- 音频时长

这样可以避免两处重复维护同一套前处理和结果归并逻辑。

### 3. 时间戳兼容策略

若用户请求：

- `--return-time-stamps`
- 或 `--subtitle-dir`

则继续保留单文件执行 + `transcribe_with_timestamp_fallback(...)` 逻辑。

原因：当前兼容逻辑依赖“某次调用失败后退回非时间戳模式”，若直接批量调用，错误定位和回退粒度会变复杂，收益也小于风险。

### 4. 视频抽音频缓存

`prepare_audio_input(...)` 生成目标 WAV 后：

- 若目标文件不存在，执行提取
- 若目标文件已存在且修改时间不早于源视频，直接复用
- 若源视频更新，则重新提取

该缓存不引入额外索引文件，仅依赖临时目录中的稳定命名。

## Error Handling

- 批量推理时保持顺序映射，若底层返回数量异常，立即抛错，避免悄悄错配结果。
- 时间戳模式继续复用现有回退逻辑，不改变错误语义。
- FFmpeg 缓存复用仅在输出文件明确可复用时生效，避免读取旧文件。

## Testing Strategy

由于仓库当前没有现成测试体系，本轮新增最小化 `unittest` 用例，覆盖共享工具层：

- 批量切片与结果顺序映射
- 时间戳请求下不走批处理
- 视频抽音频缓存命中 / 失效条件

CLI / Shell 主流程以共享函数测试为主，尽量避免引入重型端到端测试。

## Success Criteria

- 多文件、无时间戳场景下，CLI 和 Shell 均通过批量推理减少总耗时
- 时间戳与字幕导出行为不回退
- 同一源视频重复处理时不重复调用 FFmpeg
- 变更集中、可读、低耦合，为后续 GUI / API 复用打基础
